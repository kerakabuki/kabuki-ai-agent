> 守るために、変わる。

## 1. プロジェクト概要

JIKABUKI×AI は、地歌舞伎（岐阜県郡上市・気良歌舞伎）を起点に、AIの力で歌舞伎の「知る・観る・演じる」体験を豊かにするプロジェクト。

運営：気良歌舞伎（けらかぶき）
開発：気良歌舞伎事務局
技術構成：Cloudflare Workers / R2 / Dify / LINE API

---

## 2. ツインブランド構成

プロジェクトは2つのブランドで構成される。

### KABUKI PLUS+（歌舞伎を、もっと面白く。）

ターゲット：歌舞伎ファン・初心者
テーマ：歌舞伎全般を楽しむ
価値：知る・観る・学ぶ・やってみる

### JIKABUKI PLUS+（演じる人の、デジタル楽屋。）

ターゲット：地歌舞伎の演者・事務局
テーマ：地歌舞伎を発信・運営する
価値：発信する・記録する・稽古する・共有する
スケール：気良歌舞伎 → 全国約200の地歌舞伎団体へ横展開

## 3. 機能構成（命名体系）

すべて4文字の英語名で統一。

### KABUKI PLUS+

| 名前 | 役割 | 性質 | 技術 |
|------|------|------|------|
| NAVI | 読んで学ぶ | 蓄積型・静的 | R2（JSON） |
| LIVE | 今を見る | リアルタイム・動的 | Dify（自動取得） |
| RECO | 記録する | ユーザーデータ | KV / R2 |
| DOJO | やってみる | 体験型・エンタメ | YouTube API + cue JSON |

#### KABUKI NAVI ── 読んで学ぶ
- 演目ガイド（あらすじ・見どころ・配役）
- 人物事典（俳優・歴史上の人物）
- 用語いろは（歌舞伎用語をわかりやすく）
- おすすめ演目（好みに合った演目を提案）
- 学習ログ（用語クリップ・学習進捗）

#### KABUKI LIVE ── 今を見る
- 歌舞伎ニュース（毎日自動取得・歌舞伎/地歌舞伎）
- 公演スケジュール（歌舞伎座ほか公演情報・チケット）

#### KABUKI RECO ── 記録する
- 観劇ログ（観た演目・感想・推し俳優の記録）
- SNSシェア（記録内容をテキスト＋ハッシュタグで自動生成し、Web Share APIでX/LINE等に投稿）

#### KABUKI DOJO ── やってみる
- 歌舞伎クイズ（知識の腕試し）
- 台詞稽古チャレンジ（名台詞をカラオケ感覚で体験）
- 大向う道場（掛け声タイミングを音ゲー風に）

### JIKABUKI PLUS+

| 名前 | 役割 | 性質 |
|------|------|------|
| GATE | 表玄関 | 公開サイト・外向き |
| INFO | お知らせ・カレンダー | リアルタイム・動的 |
| BASE | 楽屋 | 運営ツール・内向き |
| LABO | 試す・作る | 開発・実験・稽古 |

#### JIKABUKI GATE ── 表玄関（団体を知ってもらう）
- 団体公式サイト（テンプレートで自動生成。気良歌舞伎がお手本）
- 団体チャットボット（質問に答えるだけでFAQ＋ボットが完成）
- 公演情報ページ（次回公演・過去公演を自動掲載）

#### JIKABUKI INFO ── お知らせ・カレンダー（今を知る）
- 地歌舞伎ニュース（地歌舞伎に関するニュースを自動取得）
- 地歌舞伎イベントカレンダー（全国の地歌舞伎公演・イベント一覧）

#### JIKABUKI BASE ── 楽屋（運営に使う＋業界共有データベース）

団体ごとの運営ツール：
- 公演記録・出演記録（演目・配役・日程のアーカイブ）
- 稽古メモ・参考動画（気づきの記録＋参考URLリンク）
- デジタル台本（スマホ・タブレットで稽古に使える）
- 台本共有【横展開】（団体間で台本を共有し事務局負担を軽減）

業界共有データベース（みんなで登録・みんなで使う）：
- 芝居小屋データベース（全国の芝居小屋・上演可能施設）
- 貸衣装屋データベース（衣装の取り扱い・対応演目）
- かつら師・化粧師データベース（依頼先・対応地域）
- 大道具セット所有状況（どの団体がどの演目の道具を持っているか）

#### JIKABUKI LABO ── 試す・作る（開発・実験・稽古）
- 稽古モード【実践版】（自分の役の台詞稽古・台本/動画連動）
- 台詞稽古チャレンジ【団体版】（カラオケ風の台詞練習ツール）
- 新機能ベータテスト（開発中の機能を試す場所）

## 5. けらのすけ（AI）の定義

### 一言で

「歌舞伎の友達」── 教科書ではなく、会話で寄り添う存在。

### 役割の優先順位

1. 相談相手（会話で対応）── コア
2. 案内人（サイトへ誘導）
3. サイト生成者（GATE用）
4. 解説者（知識を教える）── 最下位

### 行動原則

- まず会話で受け止める（「弁天小僧って面白い？」→「痛快な話だよ！」）
- 詳しい解説はWebに任せる（「詳しくはNAVIで読んでね→」）
- LINEでは会話のテンポを大事にする
- GATE では団体と対話しながらサイトを構築する裏方

## 6. チャネル設計

### LINE ── 会話特化

けらのすけとの対話チャネル。歌舞伎の「友達」として相談に乗り、必要に応じてWebへ案内する。

- 自然言語で歌舞伎について聞ける・相談できる
- メニュー操作も後方互換で残す（ボタン派向け）
- 詳しい情報はWebリンクで誘導

### Web ── 全機能提供

KABUKI PLUS+ と JIKABUKI PLUS+ の全機能を提供。

- NAVI / LIVE / RECO / DOJO の各ページ
- GATE / INFO / BASE / LABO の各機能
- けらのすけウィジェット（Web版チャット窓口）

## 7. 共有基盤

両ブランドで共通の技術基盤。

| 要素 | 役割 |
|------|------|
| けらのすけ | 歌舞伎の友達AI（会話エンジン） |
| 稽古エンジン | YouTube区間再生 / cue データ JSON |
| Cloudflare Workers | リクエスト受付・ルーティング・データ配信 |
| Cloudflare R2 | 静的データ（演目・用語・人物JSON、アセット） |
| Cloudflare KV | 状態管理（conversation_id、ユーザー設定） |
| Dify | LLM処理（会話AI・ニュース取得・RAG）→ Workers AI で置き換え検討中 |

### Cloudflare完結構成（検討中）

Workers AI（オープンソースLLM）を使えば、Difyへの外部依存をなくし全スタックをCloudflare内で完結できる可能性がある。月額コスト実質ゼロ。

- Workers AI 無料枠：1日10,000 Neurons（数百回の会話応答相当）

#### モデル候補と日本語力

| モデル | 日本語力 | 特徴 |
|--------|---------|------|
| Qwen3-30B-A3B（★最有力） | ◎ | アジア言語最強。MoEで軽量高速。119言語対応 |
| Llama 3.1 8B | ○ | 多言語対話に最適化。128Kコンテキスト。日本語は公式8言語外 |
| Llama 4 Scout 17B | ○ | 新世代MoE。多言語改善あり |

#### 懸念点
- Claude/GPT-4には及ばない（微妙なニュアンス・専門知識の正確さ）
- Qwen系は中国的検閲（リフューザル）傾向あり
- → けらのすけの設計でカバー：短く返す＋R2データで補う＋Webへ誘導

#### テスト計画（次回実施）
- テストモデル：Qwen3-30B-A3B / Llama 3.1 8B / Llama 4 Scout
- テスト項目：
  - 友達口調の自然さ（「弁天小僧って面白い？」）
  - 歌舞伎用語の理解（「花道って何？」）
  - インテント分類（JSON出力の精度）
  - マルチターン会話の文脈保持
  - 応答速度（LINEのUXに影響）
| LINE Messaging API | メッセージ送受信 |

## 8. けらのすけ進化計画

### 現状

「メニュー → 番号選択 → モード処理 → R2/Difyフォールバック」のFAQボット型。

### フェーズ1：脱メニュー（入口の差し替え）

既存資産を活かしつつ、入口だけを差し替える。

#### Step 1：Difyに会話フローを作る
- system prompt にけらのすけペルソナを定義
- conversation_id 対応でマルチターン会話
- R2データを参照するツール呼び出し設定

#### Step 2：Workersのルーティング変更
- 番号入力 → 従来のモード処理（後方互換）
- 自由テキスト → Difyの会話フローへ（conversation_id付き）
- KVにconversation_idをユーザーごとに保存

#### Step 3：応答の出し分け
- NAVI系 → R2データ取得 → けらのすけの言葉で返す＋Webリンク
- 雑談・相談 → Difyの会話AIがそのまま応答
- LIVE系 → ニュースAPI呼び出し
- DOJO系 → 各機能の起動リンク

### 既存資産の扱い

| 資産 | 変更 |
|------|------|
| R2データ（演目・用語・人物） | そのまま。Difyから参照される |
| Flexメッセージ生成 | そのまま。Webリンクカードとして活用 |
| ウィジェット（keranosuke-widget.js） | そのまま。Web側は変更なし |
| LINE Flexメニュー | 残す（ボタン操作派の後方互換） |
| KVモード管理 | conversation_id管理に拡張 |
| Dify | フォールバック → メインエンジンに昇格 |

### フェーズ2：RAG化（将来）

R2データをそのまま返すのではなく、LLMに通して「けらのすけの言葉」で返す。「子ども向けに教えて」「英語で教えて」のようなリクエストに対応。

### フェーズ3：GATE連動（将来）

けらのすけのエンジンを汎用化。他の地歌舞伎団体がチャットで質問に答えるだけでFAQ＋ボット＋公式サイトが自動生成される仕組みへ。

## 9. コンセプト ── 3つの柱

岐阜大学プレゼン資料より。

- 地域文化 ── 気良の地域に根付く伝統文化を未来へつなぐ
- 伝統芸能 ── 歌舞伎という「表現の力」で地域を盛り上げる
- デジタル化 ── デジタルの新しい手段で保存と発信を可能にする

この3つの交差点にJIKABUKI×AIプロジェクトがある。

## 10. ターゲット整理（メモ）

機能ごとにターゲット層が異なる。

- NAVI → 歌舞伎初心者〜中級者
- LIVE → 歌舞伎ファン全般
- RECO → 定期的に観劇する歌舞伎ファン
- DOJO → 幅広い層（バイラルポテンシャル。新規ユーザーの入口になりうる）
- GATE → 地歌舞伎に興味がある一般の人（KABUKI PLUS+のターゲットと重なる）
- INFO → 地歌舞伎ファン・関係者全般
- BASE → 地歌舞伎団体の演者・事務局
- LABO → 地歌舞伎団体の演者（稽古用）・開発者

DOJOが新規ユーザーの入口 → NAVI・LIVEへ流れる導線設計が重要。
GATEとKABUKI PLUS+のターゲットが重なる → 相互導線が必要。
